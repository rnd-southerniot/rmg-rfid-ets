## What This Is

RMG RFID ETS/PTS ‚Äî a Node/Express + Postgres backend for bundle-level RFID tracking in garment factories. ESP32 RFID stations on the factory floor scan bundles at each workstation (cutting, sewing, finishing, QC) and post events to this API. An admin UI (not in this repo) handles station mapping and monitoring.

## Commands

```bash
npm run dev          # Start dev server with hot reload (tsx watch)
npm run build        # TypeScript compile to dist/
npm run lint         # ESLint
npm test             # Run all tests (vitest)
npm run test:watch   # Vitest in watch mode

```

## Architecture

### Request flow

`server.ts` loads env/db and calls `createApp()` in `app.ts`. The app factory pattern (`createApp({ db })`) injects a `Db` (pg Pool) into all route modules, making the app testable with pg-mem.

### Route modules (`src/routes/`)

Each route file exports a factory function that takes `Db` and returns an Express Router:

- **stations.ts** ‚Äî `POST /api/v1/stations/claim`: station provisioning by MAC. Auto-creates factory if needed. Issues a `sttok_*` bearer token (SHA-256 hashed in DB).
- **events.ts** ‚Äî `POST /api/v1/events`: event ingest (requires station bearer token). Validates station is mapped (has station_id, line_id, type). Enforces station type rules (QC stations only emit QC_PASS/QC_FAIL; others only COMPLETE). Idempotent via `ON CONFLICT (station_id, event_id) DO NOTHING`. Updates bundle status on new events.
- **bundles.ts** ‚Äî `POST /api/v1/bundles`: bundle creation with RFID UID. `GET /api/v1/bundles/by-rfid/:rfidUid`: lookup.
- **admin.ts** ‚Äî CRUD for factories, lines, station mapping. Protected by `ADMIN_TOKEN` via `x-admin-token` header or `admin_token` query param.
- **adminEvents.ts** ‚Äî `GET /api/v1/admin/events/recent` and `GET /api/v1/admin/events/stream` (SSE polling).
- **stationStatus.ts** ‚Äî `POST /api/v1/station/heartbeat` and `GET /api/v1/station/me` (station self-introspection). Uses station bearer auth.

### Auth (`src/auth.ts`, `src/adminAuth.ts`)

Two auth mechanisms:
- **Station auth**: Bearer token in `Authorization` header ‚Üí SHA-256 hash lookup in `stations.token_hash`
- **Admin auth**: Static token from `ADMIN_TOKEN` env var, passed as `x-admin-token` header or `admin_token` query param

### Key modules

- `ids.ts` ‚Äî ID generation (`prefix_timestamp_random` format) and token hashing
- `validation.ts` ‚Äî Shared Zod schemas: `MacSchema` (XX:XX:XX:XX:XX:XX, auto-uppercased), `IsoTsSchema`, `EventTypeSchema` (COMPLETE | QC_PASS | QC_FAIL)
- `env.ts` ‚Äî Zod-validated env config (DATABASE_URL required, PORT defaults to 3000)
- `cors.ts` ‚Äî CORS middleware, only active when `CORS_ORIGIN` env is set

### Database

- Postgres 16 (Alpine via docker-compose, host port **5434**)
- Migrations in `db/migrations/` ‚Äî plain SQL files, applied in sort order, idempotent (`IF NOT EXISTS`)
- Tables: `factories`, `lines`, `stations`, `bundles`, `events`
- All primary keys are text IDs generated by `ulidLike()` (not actual ULIDs, format: `prefix_base36time_hex`)
- Events are append-only with unique constraint on `(station_id, event_id)` for idempotency

### Testing

- **vitest** with **pg-mem** (in-memory Postgres) ‚Äî no real database needed
- Test DB helper (`test/helpers/testDb.ts`) creates a fresh pg-mem instance and runs all migrations
- Tests create the app via `createApp({ db: pool, logLevel: 'silent' })`
- HTTP assertions use **supertest**

### Environment variables

See `.env.example`. Key vars:
- `DATABASE_URL` ‚Äî required (default: `postgres://postgres:postgres@localhost:5432/rmg_rfid_ets`)
- `ADMIN_TOKEN` ‚Äî required for admin endpoints
- `CORS_ORIGIN` ‚Äî optional, enables CORS when set

Note: docker-compose maps Postgres to host port **5434**, so local DATABASE_URL should use port 5434 unless connecting directly to the container.

### Live Demo Simulation (`scripts/simulateDemo.ts`)

Continuous simulation against a running backend with real mapped stations. Used for CEO demos and Admin UI testing.

- **Startup**: discovers mapped stations via admin API, re-claims for fresh tokens, builds pipeline ordered by type (cutting ‚Üí sewing ‚Üí finishing ‚Üí qc)
- **Bundle flow**: creates bundles with 7-byte RFID UIDs, flows through all stations with delays
- **Scenarios**: 70% QC Pass, 20% QC Fail (with defect codes), 10% Rework (fail ‚Üí re-sew ‚Üí pass)
- **Heartbeats**: background POST every 30s to keep `last_seen_at` fresh
- **Speed**: `SIM_SPEED=realistic` (30-60s, default) or `SIM_SPEED=fast` (5-10s)
- **Requires**: running backend, Postgres, `ADMIN_TOKEN` in `.env`, at least 2 mapped stations for `SEED_FACTORY_CODE`
